
1	# mihomo-webui 预设策略组与 proxy-provider 问题复盘
 
2	
 
3	## 背景
 
4	
 
5	mihomo-webui 提供了“一键应用策略组/分流模式”的能力，例如：
 
6	
 
7	- 极简模式
 
8	- 智能容灾
 
9	- 日常模式
 
10	- 流媒体模式
 
11	- 游戏加速模式
 
12	- 高可用模式
 
13	
 
14	这些模式会自动生成 `proxy-groups`，其中部分策略组依赖节点池，例如：
 
15	
 
16	```yaml
 
17	proxy-groups:
 
18	  - name: AUTO-GLOBAL
 
19	    type: url-test
 
20	    use:
 
21	      - rta
 
22	```
 
23	
 
24	这里的 `use` 必须引用 mihomo 配置中真实存在的 `proxy-providers`。
 
25	
 
26	初次安装 mihomo 时，mihomo 本身不会自动创建任何 `proxy-providers`。节点来源通常需要用户通过以下方式提供：
 
27	
 
28	- 手动在配置文件中写入 `proxies`
 
29	- 手动配置 `proxy-providers`
 
30	- 在 mihomo-webui 中添加订阅并生成 provider
 
31	
 
32	## 问题现象
 
33	
 
34	在没有可用 provider，或 provider 未被正确识别时，mihomo-webui 的预设策略组可能生成了不合理配置，例如：
 
35	
 
36	```yaml
 
37	proxy-groups:
 
38	  - name: AUTO-GLOBAL
 
39	    type: url-test
 
40	    use: []
 
41	```
 
42	
 
43	或者保留了 webui 内部模板占位符：
 
44	
 
45	```yaml
 
46	proxy-groups:
 
47	  - name: AUTO-GLOBAL
 
48	    type: url-test
 
49	    use:
 
50	      - all
 
51	```
 
52	
 
53	其中：
 
54	
 
55	- `use: []` 表示策略组没有任何节点来源。
 
56	- `all` 是 webui 模板中的“所有 provider”占位符，不是 mihomo 原生一定存在的 provider 名。
 
57	- mihomo 只能识别真实存在的 provider 名，不能识别 webui 的抽象占位符。
 
58	
 
59	因此该配置可能导致：
 
60	
 
61	- mihomo reload 失败
 
62	- 策略组无可用节点
 
63	- 规则命中后没有可用出口
 
64	- 用户表现为“断网”或“代理不可用”
 
65	
 
66	## 2026-05-09 系统问题复盘
 
67	
 
68	### 问题摘要
 
69	
 
70	系统在应用 mihomo-webui 预设策略组时，未充分校验当前 mihomo 配置中是否存在可用 `proxy-providers`，导致生成的 `proxy-groups` 可能缺少有效节点来源。
 
71	
 
72	该问题属于 webui 配置生成与配置校验链路缺陷：
 
73	
 
74	- 预设策略组模板中使用了 `all` 表示“所有 provider”。
 
75	- 写入 mihomo 配置前，系统没有在所有路径中保证 `all` 被展开为真实 provider 名。
 
76	- 当 provider 为空或未识别时，系统仍可能生成依赖 provider 的策略组。
 
77	- 生成结果可能出现 `use: []` 或无有效节点来源的 `url-test` / `fallback` 策略组。
 
78	
 
79	### 影响范围
 
80	
 
81	受影响场景：
 
82	
 
83	- 初次安装后尚未添加订阅/provider。
 
84	- 配置文件中 `proxy-providers` 为空。
 
85	- provider 已存在但 webui 未正确读取或展开 provider 名。
 
86	- 用户应用“智能容灾”“极简模式”“日常模式”等依赖节点池的预设策略组。
 
87	
 
88	可能影响：
 
89	
 
90	- mihomo reload 配置失败。
 
91	- 策略组存在但没有可用节点。
 
92	- 规则命中后无法选出有效出口。
 
93	- 代理不可用，表现为网络访问异常。
 
94	
 
95	### 触发条件
 
96	
 
97	满足以下条件时可能触发：
 
98	
 
99	1. 当前 mihomo 配置没有可用 `proxy-providers`，或 webui 没有正确识别 provider。
 
100	2. 用户应用包含 `url-test`、`fallback`、`load-balance` 等依赖节点池的预设策略组。
 
101	3. webui 将模板中的 `use: ['all']` 写入配置时未正确展开，或展开结果为空。
 
102	
 
103	### 系统缺陷
 
104	
 
105	本次问题暴露出以下系统缺陷：
 
106	
 
107	1. **预设策略组依赖前置条件不明确**
 
108	
 
109	   预设模式依赖节点来源，但 UI 没有强制要求先添加订阅/provider。
 
110	
 
111	2. **模板占位符泄漏到配置层**
 
112	
 
113	   `all` 是 webui 内部模板概念，不应直接进入 mihomo 配置。
 
114	
 
115	3. **无 provider 场景缺少产品级拦截**
 
116	
 
117	   系统没有在应用预设模式前明确判断 provider 是否为空，并给出“请先添加订阅”的引导。
 
118	
 
119	4. **后端防御不足**
 
120	
 
121	   API 保存策略组时原本没有统一归一化逻辑，无法兜底修正空 `use`、未展开 `all`、无节点来源等异常输入。
 
122	
 
123	5. **配置校验提示不够强**
 
124	
 
125	   对“策略组没有可用节点来源”的问题缺少阻断或高优先级提示，用户可能在配置不可用时仍继续应用。
 
126	
 
127	### 已采取修复
 
128	
 
129	已完成以下系统修复：
 
130	
 
131	1. 前端应用预设策略组时，将 `use: ['all']` 展开为当前真实 provider 名。
 
132	2. 当前没有 provider 时，不再生成 `use: []`，而是使用 `DIRECT` 作为安全兜底。
 
133	3. 后端保存策略组时增加统一归一化逻辑：
 
134	   - 清理空字符串。
 
135	   - 展开 `all` 为真实 provider。
 
136	   - 无节点来源时补充 `DIRECT`。
 
137	4. 修复阻塞构建的 VLESS URI 名称格式化问题。
 
138	5. 已通过：
 
139	   - `go test ./...`
 
140	   - `web` 前端 `npm run build`
 
141	
 
142	### 后续系统改进
 
143	
 
144	后续应继续完善以下系统能力：
 
145	
 
146	1. **无 provider 时拦截高级预设模式**
 
147	
 
148	   当没有订阅/provider 时，不应直接应用依赖节点池的模式，应提示用户先添加订阅。
 
149	
 
150	2. **增加初次安装引导**
 
151	
 
152	   推荐流程：添加订阅 → 拉取 provider → 校验节点数量 → 选择预设策略组 → 应用配置。
 
153	
 
154	3. **增强配置校验**
 
155	
 
156	   对以下情况给出明确错误或高优先级警告：
 
157	
 
158	   - `use: []`
 
159	   - `use` 引用不存在的 provider
 
160	   - `url-test` / `fallback` / `load-balance` 无任何节点来源
 
161	
 
162	4. **预设策略组应用前展示变更预览**
 
163	
 
164	   在真正写入配置前展示将要生成的 `proxy-groups`，并标明使用了哪些 provider。
 
165	
 
166	5. **保持后端兜底长期存在**
 
167	
 
168	   即使前端增加拦截，后端仍必须防止 API 被直接调用时写入非法策略组。
 
169	
 
170	## 根因
 
171	
 
172	根因不是 mihomo 初装没有 provider。
 
173	
 
174	更准确地说：
 
175	
 
176	> mihomo-webui 的预设策略组生成逻辑错误地假设当前配置已经存在可用 provider，或者错误地把内部占位符写入了 mihomo 配置。
 
177	
 
178	mihomo 的责任是校验并运行合法配置；mihomo 不应该为了 webui 的模板占位符做兼容。
 
179	
 
180	webui 的责任是：
 
181	
 
182	1. 读取当前 mihomo 配置能力。
 
183	2. 判断是否存在可用节点来源。
 
184	3. 只生成 mihomo 能识别、能运行的合法配置。
 
185	
 
186	## 不合理设计点
 
187	
 
188	原设计隐含了一个错误前提：
 
189	
 
190	> webui 预设模式可以直接生成依赖 provider 的策略组，即使用户尚未添加订阅/provider。
 
191	
 
192	这会导致 webui 把产品层面的“所有订阅/所有节点”概念直接泄漏到 mihomo 配置层。
 
193	
 
194	合理的边界应该是：
 
195	
 
196	- webui 可以有“所有 provider”这种 UI/模板概念。
 
197	- 但写入 mihomo 配置前，必须解析成真实 provider 名。
 
198	- 如果没有真实 provider，就不能写 `use: []` 或 `use: [all]`。
 
199

261	## 当前修复的局限
 
262	
 
263	当前修复可以避免写坏配置，但仍然只是“安全兜底”。
 
264	
 
265	当初次安装没有 provider 时，使用 `DIRECT` 兜底意味着：
 
266	
 
267	- 配置是合法的。
 
268	- mihomo 不会因为空策略组失败。
 
269	- 但用户实际上没有代理节点。
 
270	- 用户可能误以为已经完成代理配置。
 
271	
 
272	因此，`DIRECT` 兜底适合防止断网和配置损坏，但不是最佳产品体验。
 
273	
 
274	## 推荐产品方案
 
275	
 
276	### 方案 A：无 provider 时禁止应用依赖节点池的模式（推荐）
 
277	
 
278	当 `proxy-providers` 为空，且当前模式包含 `url-test`、`fallback`、`load-balance` 等依赖节点池的组时：
 
279	
 
280	- 禁用模式按钮；或
 
281	- 点击时弹窗提示：
 
282	
 
283	```text
 
284	当前没有可用订阅/provider。
 
285	请先添加订阅或导入节点，再应用该策略组模式。
 
286	```
 
287	
 
288	优点：
 
289	
 
290	- 最符合用户预期。
 
291	- 不会让用户误以为已经配置好代理。
 
292	- 不会生成“看似完整但实际 DIRECT”的配置。
 
293	
 
294	缺点：
 
295	
 
296	- 初次安装时需要多一步引导。
 
297	
 
298	### 方案 B：提供初次安装引导
 
299	
 
300	首次进入系统配置时，引导流程改为：
 
301	
 
302	1. 添加订阅 URL。
 
303	2. 拉取并校验 provider。
 
304	3. 显示节点数量。
 
305	4. 选择预设策略组模式。
 
306	5. 应用配置并 reload mihomo。
 
307	
 
308	优点：
 
309	
 
310	- 产品体验完整。
 
311	- 用户不需要理解 provider/proxy-groups 细节。
 
312	
 
313	缺点：
 
314	
 
315	- 实现成本较高。
 
316	
 
317	### 方案 C：保留 DIRECT 安全模式，但明确提示
 
318	
 
319	如果用户没有 provider 仍然应用模式，则允许写入 `DIRECT` 兜底配置，但必须明确提示：
 
320	
 
321	```text
 
322	当前没有订阅/provider，本次只应用 DIRECT 安全配置。
 
323	该配置不会提供代理能力，请添加订阅后重新应用策略组。
 
324	```
 
325	
 
326	优点：
 
327	
 
328	- 不会写坏配置。
 
329	- 允许用户先完成基础配置。
 
330	
 
331	缺点：
 
332	
 
333	- 仍可能让用户困惑为什么不能代理。
 
334	
 
335	## 建议落地顺序
 
336	
 
337	1. 保留当前后端兜底，防止任何入口写坏配置。
 
338	2. 前端增加 provider 检查：无 provider 时禁用或拦截高级模式。
 
339	3. 添加“先添加订阅”的引导入口。
 
340	4. 在策略组应用结果中展示本次使用了哪些 provider。
 
341	5. 在配置校验页增加更明确的问题提示：
 
342	
 
343	```text
 
344	策略组没有可用节点来源，请添加 proxies、use provider，或使用 DIRECT 兜底。
 
345	```

347	## 验收标准
 
348	
 
349	### 无 provider 场景
 
350	
 
351	- 初次安装、`proxy-providers` 为空时，点击预设模式不会写出：
 
352	
 
353	```yaml
 
354	use: []
 
355	```
 
356	
 
357	- 不会写出：
 
358	
 
359	```yaml
 
360	use:
 
361	  - all
 
362	```
 
363	
 
364	- mihomo reload 不应因为空策略组失败。
 
365	
 
366	- UI 必须提示用户当前没有代理节点来源。
 
367	
 
368	### 有 provider 场景
 
369	
 
370	- 假设配置中存在：
 
371	
 
372	```yaml
 
373	proxy-providers:
 
374	  rta:
 
375	    type: http
 
376	    url: https://example.com/sub.yaml
 
377	```
 
378	
 
379	- 应用预设模式后，相关策略组应写为：
 
380	
 
381	```yaml
 
382	use:
 
383	  - rta
 
384	```
 
385	
 
386	- 不应保留 `all` 占位符。
 
387	
 
388	### 后端防御
 
389	
 
390	- 直接调用 API 保存以下 payload：
 
391	
 
392	```json
 
393	{
 
394	  "name": "AUTO",
 
395	  "type": "url-test",
 
396	  "proxies": [],
 
397	  "use": ["all"]
 
398	}
 
399	```
 
400	
 
401	后端必须：
 
402	
 
403	- 有 provider 时展开为真实 provider。
 
404	- 无 provider 时补充 `DIRECT`。
 
405	
 
406	## 结论
 
407	
 
408	这个问题本质上是 webui 配置生成逻辑的问题，不是 mihomo 初次安装的问题。
 
409	
 
410	mihomo-webui 不应要求 mihomo 支持 webui 的预设占位符；相反，webui 必须把预设模式转换成 mihomo 原生合法配置。
 
411	
 
412	当前修复已经避免写出会导致断网/不可用的空策略组。后续应继续完善初次安装引导和无 provider 时的 UI 拦截，避免用户误操作。
